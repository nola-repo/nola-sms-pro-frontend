# Backend Handoff: Fixing GHL Contacts 401 Unauthorized Issue

**Date:** April 2026
**Component:** GHL API Authentication & Refresh Tokens
**Target Audience:** NOLA SMS Pro Backend (PHP) Team

## Overview
The frontend is receiving a `401 Unauthorized` error when attempting to fetch GHL Contacts (`/api/ghl-contacts`). This issue is caused by a **credential mismatch** during the OAuth token refresh process.

### The Core Problem
NOLA SMS Pro currently utilizes two distinct GHL Marketplace Apps:
1. **Agency App**: 
   - `GHL_CLIENT_ID`: `69d31f33b3071b25dbcc5656-mnqxvtt3`
   - `GHL_CLIENT_SECRET`: `64b90a28-8cb1-4a44-8212-0a8f3f255322`
2. **User App (Sub-accounts)**: 
   - `CLIENT_ID`: `6999da2b8f278296d95f7274-mm9wv85e`
   - `CLIENT_SECRET`: `dfc4380f-6132-49b3-8246-92e14f55ee78`

When a sub-account is authorized, its token is generated using the **User App**. However, when that token inevitably expires, the backend API attempts to refresh it using the **Agency App** credentials (because the PHP code relies on a statically injected `getenv('GHL_CLIENT_ID')` that currently maps to the Agency App).

Since the refresh token was generated by the User App, GHL rejects the refresh attempt using the Agency App's Client ID with a `401 Unauthorized` error.

---

## Action Plan: Fixing the Mismatch

To permanently resolve this issue, you must update the backend to dynamically select the correct `CLIENT_ID` and `SECRET` based on which app generated the original token.

### 1. Update Environment Variables
Update your Cloud Run deployment environment (and local `.env`) to explicitly provide both pairs of credentials.

```env
# Agency App Credentials
GHL_AGENCY_CLIENT_ID="69d31f33b3071b25dbcc5656-mnqxvtt3"
GHL_AGENCY_CLIENT_SECRET="64b90a28-8cb1-4a44-8212-0a8f3f255322"

# User App (Subaccount) Credentials
GHL_USER_CLIENT_ID="6999da2b8f278296d95f7274-mm9wv85e"
GHL_USER_CLIENT_SECRET="dfc4380f-6132-49b3-8246-92e14f55ee78"
```

### 2. Update Token Registration (`webhook/ghl_oauth.php`)
When a new token is generated during the OAuth callback, store the `client_id` (or `app_type`) that was used inside the Firestore `integrations` document. This allows the backend to "remember" which credentials to use for future refreshes.

**File:** `api/webhook/ghl_oauth.php` (and similar exchange endpoints)
```php
// Use the appropriate client ID based on the callback context
$clientId = $_GET['app_type'] === 'agency' ? getenv('GHL_AGENCY_CLIENT_ID') : getenv('GHL_USER_CLIENT_ID');
// ... 
$db->collection('integrations')
    ->document($docId)
    ->set([
        'access_token' => $result['access_token'],
        'refresh_token' => $result['refresh_token'],
        // ... existing fields ... //
        'client_id' => $clientId // <--- IMPORTANT: Store the client ID
    ], ['merge' => true]);
```

### 3. Update Token Refresh Logic
Modify the `refreshGHLToken()` logic to read the stored `client_id` from the token's Firestore document. If none is found (legacy tokens), you should ideally attempt the refresh with the User App creds first, then fallback to Agency. 

You need to update this logic everywhere a refresh happens. This includes:
- `api/ghl-contacts.php` 
- `api/ghl_contacts.php`
- `api/services/GhlClient.php`

**Example Fix for `services/GhlClient.php`:**
```php
private function refreshToken(): void
{
    // Retrieve the client_id that generated this token
    $storedClientId = $this->integration['client_id'] ?? null;
    
    // Choose the correct credentials
    if ($storedClientId === getenv('GHL_AGENCY_CLIENT_ID')) {
        $clientId = getenv('GHL_AGENCY_CLIENT_ID');
        $clientSecret = getenv('GHL_AGENCY_CLIENT_SECRET');
    } else {
        // Default to User App for legacy/unknown tokens
        $clientId = getenv('GHL_USER_CLIENT_ID');
        $clientSecret = getenv('GHL_USER_CLIENT_SECRET');
    }

    $refreshToken = $this->integration['refresh_token'] ?? null;
    
    // ... complete the refresh request using $clientId and $clientSecret ... //
}
```

Once these changes are deployed to Google Cloud Run, the `GhlClient` will correctly match the Refresh Token to the appropriate Application Client ID, successfully refreshing the token and resolving the `401 Unauthorized` errors.
